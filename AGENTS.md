# AGENTS.md

## 项目目标
本仓库是一个 Godot 4 极简禅意 3D 高尔夫原型。

项目的主要目的不是做完整产品，而是验证一件事：
程序化生成器能否稳定生成平静、可读、可玩的单洞高尔夫地形。

核心验证点：
- 程序化单洞生成
- 地形可读性
- 击球手感
- 镜头稳定与平静感
- 入洞后连续生成下一洞的节奏

## 设计哲学
整体方向必须保持：
- 极简
- 禅意
- 安静
- 平静
- 可读
- 稀疏
- 低 UI
- 低机制
- 低视觉噪音

好的洞应该像短小的空间诗，而不是挑战地图。

一个好的生成洞应当感觉：
- 平静
- 清晰
- 可读
- 宽松
- 轻微变化
- 坡度温和
- 视觉干净
- 情绪安静

一个坏的生成洞通常是：
- 嘈杂
- 破碎
- 太陡
- 太窄
- 太绕
- 太混乱
- 太装饰化
- 太“游戏设计”
- 太系统化

## 这不是
本项目不是：
- 功能丰富的高尔夫游戏
- 系统复杂的 roguelike
- 完整商业产品
- 大型生产架构
- 框架项目
- 未来可扩展内容平台

不要为了未来假想需求建立复杂系统。
不要用架构证明能力。
要用平静的地形、可读的洞和小范围内好玩的原型证明方向。

## 硬性禁止
不要添加：
- 商店
- 升级
- 成就
- 解锁树
- 多球杆
- 风
- 长期进度系统
- 分数冲刺元系统
- 复杂 HUD
- 重型菜单系统
- 装饰性过度设计
- 大量未来扩展接口
- 不必要的 manager、service、registry、factory、plugin system
- ECS
- 事件总线，除非绝对必要
- 服务定位器
- 通用内容数据库
- 程序化生物群系系统
- 装饰物生成系统
- 复杂编辑器工具
- 高级存档系统

当在以下选项之间选择时：
- 更干净的架构 vs 更快的可玩迭代：选择更快的可玩迭代。
- 更多功能 vs 更好的地形感觉：选择更好的地形感觉。
- 真实模拟 vs 平静可控的手感：选择平静可控的手感。
- 新奇机制 vs 可读与连续节奏：选择可读与连续节奏。

## 玩法约束
玩法循环必须保持极简：

生成一洞 -> 击球 -> 计杆 -> 入洞 -> 下一洞

只保留这个循环需要的内容。除非明确要求，不要扩展玩法系统。

## 程序化生成哲学
生成器必须先生成空间骨架，再生成地形。

不要从随机噪声地形开始，再把球洞放上去。

推荐流程：
1. 选择洞型。
2. 生成发球点和洞杯位置。
3. 生成简单的 2D 路径或空间骨架。
4. 从空间骨架生成高度场。
5. 抚平发球区。
6. 抚平洞杯区。
7. 添加极少量表面噪声。
8. 验证可玩性。
9. 丢弃坏洞并重新生成。

生成器应优先考虑轮廓、可读性和平静感，而不是真实地貌或最大新奇度。

## 允许的洞型
只使用很小的一组洞型：
- 直视型
- 轻微弯曲型
- 高台型
- 远景型
- 盆地型

洞型之间的变化应来自细微参数差异，而不是新机制或噱头。

## 必要生成参数
生成系统至少应围绕这些参数变化：
- `holeLength`
- `widthBase`
- `startHeight`
- `endHeight`
- `slopeStrength`
- `curveAmount`
- `holePlatformRadius`
- `startPlatformRadius`
- `edgeFalloff`
- `noiseAmplitude`
- `calmness`

这些参数应服务于平静、可读、连续的空间节奏。

## 验证规则
生成洞只有满足以下条件才可接受：
- 发球区稳定且可读。
- 洞杯区坡度不过大。
- 从发球点到洞杯有清晰空间方向。
- 球道宽度不会窄到令人烦躁。
- 整体坡度保持温和。
- 至少一种或多种标准击球力度能产生合理运动。
- 画面感觉平静，而不是混乱。

应拒绝以下洞：
- 太陡
- 太窄
- 太吵
- 太随机
- 太破碎
- 视觉不清
- 洞杯附近令人沮丧

## 序列生成规则
不要把每一洞都当成完全独立的随机地图。

短序列应有细微节奏。变化应主要来自：
- 略微不同的长度
- 略微不同的高差
- 略微不同的开阔程度
- 略微不同的曲率

不要通过复杂机制、装饰物或噱头制造变化。

## 视觉规则
视觉方向可以在“极简禅意”和“克制梦幻赛博”之间取平衡。

允许：
- 低多边形
- 干净形状
- 克制的赛博网格线
- 梦幻但低噪音的青、紫、粉色彩
- 根据地块坡度和朝向进行颜色变化
- 球、洞杯、地面之间有明确对比
- 地形看起来有意图，而不是随机

避免：
- 强烈霓虹轰炸
- 复杂后处理
- 杂乱装饰物
- 用视觉噪音掩盖生成质量
- 让画面热闹到破坏可读性

## UI 规则
项目应几乎没有 UI。

只允许最小 UI：
- 杆数
- 下一洞过渡
- 可选的重开快捷键

除此之外，除非明确要求，不要添加菜单、面板、统计页、解锁页或复杂 HUD。

## 镜头规则
镜头必须：
- 平静
- 稳定
- 可读
- 帮助玩家理解地形形状

避免：
- 戏剧性镜头运动
- 频繁切镜
- 夸张震动
- 过度电影化

## 物理规则
物理方向：
- 不追求真实高尔夫模拟
- 优先稳定和可读
- 坡面互动温和
- 阻尼可预测
- 不做复杂旋转系统
- 手感优先于真实性

## 工程规则
保持项目小。

目标规模：
- 一个主可玩场景
- 一个程序化球洞生成器
- 一个球洞验证器
- 一个简单游戏控制器
- 一个球控制或物理脚本
- 一个镜头 rig
- 一个极简 UI 脚本

工程约束：
- 脚本数量保持低。
- 场景数量保持低。
- 文件夹结构保持简单。
- 不要提前过度模块化。
- 不要创建“未来可能用到”的抽象。
- 不要创建大型企业式代码结构。
- 优化快速手感迭代，而不是架构纯度。

当不确定时，选择更简单的实现。

## 当前阶段完成定义
未来可玩原型的完成标准：
- 能从 seed 生成球洞。
- 生成洞可玩。
- 杆数统计可用。
- 入洞完成逻辑可用。
- 能进入下一洞。
- 生成洞视觉上有意图且平静。
- 整体感觉更接近禅意高尔夫，而不是普通程序化测试地图。

## 给后续 Codex 的提醒
开始工作前先阅读本文件。

如果用户没有明确要求扩展范围，默认保持小范围实现。

不要用复杂架构打动用户。
用平静地形、可读球洞和小而可玩的原型打动用户。
